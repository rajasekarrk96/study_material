XPath Best Practices Guide

1️⃣ Absolute vs Relative XPath

❌ Avoid Absolute XPath:
/html/body/div/form/input
Breaks easily when DOM structure changes!
✅ Use Relative XPath:
//input[@id='username']
Starts with // to search from any level

2️⃣ Locator Priority (Best to Worst)

Priority Type Stability Performance
1 ID Highest ⚡ Fastest
2 Name High ⚡ Fast
3 CSS Class Medium ⚡ Fast
4 XPath with attributes Medium Medium
5 Dynamic XPath Low Slow

3️⃣ When to Use XPath vs CSS

✅ Use XPath when:
  • Need to traverse UP the DOM (parent, ancestor)
  • Locating by text content
  • Complex logic and functions needed
  • Multiple conditions required
✅ Use CSS Selector when:
  • Simple attribute-based selection
  • Performance is critical
  • Only traversing DOWN the DOM
  • Modern browser compatibility needed

4️⃣ Performance Optimization

❌ Slow XPath (searches entire DOM):
//button[contains(text(), 'Submit')]
✅ Faster XPath (specific scope):
//form[@name='login-form']//button[contains(text(), 'Submit')]
⚠️ Start XPath from most specific parent, then navigate to target

5️⃣ Building Robust Locators

❌ Fragile (breaks on any change):
//button[2]
//span[@class='btn-submit']
✅ Robust (tolerates changes):
//button[contains(@class, 'submit')]
//button[starts-with(@name, 'btn')]
//button[contains(normalize-space(), 'Submit')]

6️⃣ Debug XPath in Chrome DevTools

// In Chrome Console, test XPath instantly:
$x("//button[@name='submit-btn']")

// Returns array of matched elements
// If empty, your XPath has no matches
