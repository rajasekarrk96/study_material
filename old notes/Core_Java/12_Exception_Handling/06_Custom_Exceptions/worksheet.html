Question 1: Custom Exception Classes

Problem Statement

Write a Java program that demonstrates the use of custom exception classes to handle application-specific exceptions. Create a custom exception by extending the Exception class.

Solution

class MyException extends Exception { MyException(String msg) { super(msg); } } class _04_ { public static void main(String[] args) { try { throw new MyException("Custom Exception Occurred"); } catch (MyException e) { System.out.println(e.getMessage()); } } }

Sample Output

Custom Exception Occurred
Key Concepts:
  • Custom Exception: Extend Exception class for checked exceptions
  • super(msg): Pass message to parent Exception class
  • Application-Specific: Create meaningful exception names
  • getMessage(): Retrieve exception message
Best Practices: Use descriptive names like InvalidAgeException, InsufficientFundsException

Question 2: try-with-resources Statement

Problem Statement

Write a Java program that demonstrates the use of the try-with-resources statement to automatically close resources. Show how resources implementing AutoCloseable are automatically closed.

Solution

import java.io.*; class _05_ { public static void main(String[] args) { try (FileReader fr = new FileReader("test.txt")) { System.out.println("File opened"); } catch (Exception e) { System.out.println("Exception handled"); } } }

Sample Output

Exception handled (if file doesn't exist) or File opened (if file exists, then automatically closed)

How try-with-resources Works

// Traditional approach (manual closing) FileReader fr = null; try { fr = new FileReader("test.txt"); // Use file } catch (Exception e) { e.printStackTrace(); } finally { if (fr != null) { try { fr.close(); } catch (IOException e) { e.printStackTrace(); } } } // try-with-resources (automatic closing) try (FileReader fr = new FileReader("test.txt")) { // Use file } catch (Exception e) { e.printStackTrace(); } // fr.close() called automatically!
Key Concepts:
  • AutoCloseable: Resources must implement AutoCloseable or Closeable
  • Automatic Closing: close() called automatically after try block
  • Multiple Resources: Can declare multiple resources separated by semicolons
  • Cleaner Code: No need for explicit finally block
Example with Multiple Resources: try (FileReader fr = new FileReader("in.txt"); FileWriter fw = new FileWriter("out.txt")) { ... }

Question 3: assert Statement

Problem Statement

Write a Java program that demonstrates the use of the assert statement to handle exceptions. Show how assertions can be used to validate assumptions during development.

Solution

class _06_ { public static void main(String[] args) { int age = 15; assert age >= 18 : "Age must be 18 or above"; System.out.println("Valid age"); } }

Sample Output

Valid age (if assertions are disabled - default) or Exception in thread "main" java.lang.AssertionError: Age must be 18 or above (if assertions are enabled with -ea flag)

Enabling Assertions

// Compile javac _06_.java // Run without assertions (default) java _06_ // Output: Valid age // Run with assertions enabled java -ea _06_ // Output: AssertionError: Age must be 18 or above // Enable for specific class java -ea:_06_ _06_ // Enable for package java -ea:com.mypackage... _06_
Key Concepts:
  • assert: Tests a boolean condition
  • AssertionError: Thrown when assertion fails
  • Disabled by Default: Must enable with -ea flag
  • Development Tool: Use for debugging, not production error handling
When to Use Assertions:
  • Internal invariants (conditions that should always be true)
  • Pre-conditions and post-conditions
  • Control flow invariants
When NOT to Use:
  • Validating user input (use exceptions instead)
  • Production error handling
  • Side effects (code that must execute)

Question 4: Exception Chaining

Problem Statement

Write a Java program that demonstrates the use of the exception chaining mechanism to propagate exceptions across multiple layers of an application. Show how to preserve the original exception while throwing a new one.

Solution

class _10_ { static void method1() { try { method2(); } catch (Exception e) { throw new RuntimeException("Exception in method1", e); } } static void method2() throws Exception { throw new Exception("Original Exception"); } public static void main(String[] args) { try { method1(); } catch (Exception e) { e.printStackTrace(); } } }

Sample Output

java.lang.RuntimeException: Exception in method1 at _10_.method1(_10_.java:7) at _10_.main(_10_.java:16) Caused by: java.lang.Exception: Original Exception at _10_.method2(_10_.java:12) at _10_.method1(_10_.java:5) ... 1 more

Understanding Exception Chaining

// Without chaining (loses original exception) try { method2(); } catch (Exception e) { throw new RuntimeException("Exception in method1"); // Original exception lost! } // With chaining (preserves original exception) try { method2(); } catch (Exception e) { throw new RuntimeException("Exception in method1", e); // Original exception preserved as "cause" } // Accessing the cause try { method1(); } catch (Exception e) { System.out.println("Current: " + e.getMessage()); System.out.println("Cause: " + e.getCause().getMessage()); }
Key Concepts:
  • Exception Chaining: Link exceptions in a cause-and-effect chain
  • Caused by: Shows the root cause in stack trace
  • getCause(): Retrieve the underlying exception
  • Layered Architecture: Each layer can add context
Benefits:
  • Preserves original exception information
  • Adds context at each layer
  • Complete stack trace for debugging
  • Better error diagnosis
Real-World Example:
  • Database layer: SQLException
  • Service layer: DataAccessException(cause: SQLException)
  • Controller layer: ServiceException(cause: DataAccessException)
