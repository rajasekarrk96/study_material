08. Selenium Locators

📖 Introduction

"If you can't identify it, you can't automate it." Locators are the fundamental building blocks of any Selenium script. They are the addresses that tell Selenium exactly which HTML element (button, textbox, link) you want to interact with on the web page.

💡 What You'll Learn

  • The 8 different types of locators in Selenium
  • When to use which locator strategy
  • The difference between Absolute and Relative XPath
  • Why CSS Selectors are often preferred over XPath

🔍 Detailed Explanation

The 8 Locator Strategies

Selenium provides 8 mechanisms to find elements, typically using the By class:

  1. ID: The Gold Standard. Fast, unique, and reliable. Always use this if available.
  2. Name: Good alternative if ID is missing.
  3. ClassName: Useful for grouping elements, but be careful of duplicates.
  4. TagName: Best for finding groups of elements (e.g., all <a> tags).
  5. LinkText: Finds hyperlinks by their exact visible text.
  6. PartialLinkText: Finds hyperlinks by a portion of their visible text.
  7. CSS Selector: The "Swiss Army Knife". Fast and supports complex relationships.
  8. XPath: The "Nuclear Option". Most powerful, can traverse up the DOM, but can be slower and brittle if not reliable.

XPath vs. CSS Selector

This is a classic debate. CSS Selectors are generally faster and have cleaner syntax. XPath is necessary when you need to navigate UP the DOM (from child to parent) or locate by text content, which CSS cannot do.

💻 Code Examples

Here is how you use each locator strategy in Java:

Locator Examples

// 1. By ID (Best)
driver.findElement(By.id("username"));

// 2. By Name
driver.findElement(By.name("email_address"));

// 3. By ClassName
driver.findElement(By.className("login-btn"));

// 4. By CSS Selector (ID)
driver.findElement(By.cssSelector("#username"));

// 5. By CSS Selector (Class)
driver.findElement(By.cssSelector(".login-btn"));

// 6. By XPath (Relative)
driver.findElement(By.xpath("//input[@id='username']"));

// 7. By LinkText
driver.findElement(By.linkText("Forgot Password?"));

⚠️ Common Pitfalls

Avoid Absolute XPath: Never use XPaths like /html/body/div[1]/div[2]/form/input. If the developer adds one div at the top, your test breaks. Always use Relative XPath (starting with //).

🎯 Real-World Scenario

Practice Scenario

Target URL: https://www.facebook.com

Goal: Identify the correct locators for the Login Form.

  1. Locate the "Email" field using ID.
  2. Locate the "Password" field using Name.
  3. Locate the "Log In" button using CSS Selector.
  4. Locate the "Forgotten password?" link using LinkText.
  5. Locate the "Create new account" button using XPath.

📝 Interview Questions

Q1: Which is faster, XPath or CSS Selector?

Answer: CSS Selector is technically faster because most browsers implement it natively. XPath engines can be slightly slower, but in modern browsers, the difference is negligible for small test suites.

Q2: How do you handle dynamic elements (ids that change)?

Answer: We cannot use the full ID. We use strategies like contains(), starts-with(), or locate by text, or locate a stable parent and traverse down.
